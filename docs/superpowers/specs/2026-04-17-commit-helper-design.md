# Commit Helper 技能设计方案

**日期：** 2026-04-17  
**状态：** 对话中已确认，等待书面方案复核  
**项目：** `fuxiaochen`

## 1. 背景

当前仓库已经采用 Conventional Commits 规范，但在实际提交流程中仍然存在几个问题：

- 提交信息经常需要临时人工组织，风格不稳定
- 中文语义总结容易写得过长、过泛或不准确
- 工作区中若混有多个主题的改动，容易被一次性错误提交
- 当用户希望“分析改动并自动提交”时，缺少一个稳定、可复用的工作流

为了解决这些问题，需要创建一个仓库内可复用的 skill，用来分析当前仓库改动、生成中文 Conventional Commits 提交信息，并在满足条件时自动完成提交。

## 2. 目标

这个 skill 的目标是提供一个**受控的自动提交助手**，能力边界如下：

- 分析当前仓库工作区改动
- 判断改动是否大致属于单一主题
- 为单一主题改动生成一条中文 Conventional Commits 提交信息
- 自动执行 `git add` 与 `git commit`
- 在发现混杂改动时停止自动提交，并输出拆分建议

这个 skill 必须优先保证提交安全性和语义准确性，而不是追求“无论什么情况都能自动提交”。

## 3. 非目标

本次 skill 明确不负责以下能力：

- 不自动拆分成多个 commit 并逐个提交
- 不自动挑选部分文件组成提交
- 不自动执行 `git push`
- 不自动生成长篇 commit body
- 不自动处理复杂的 `rebase`、`merge`、`cherry-pick` 等特殊 git 状态
- 不在混杂改动时替用户做主观裁剪

## 4. 技能定位

### 4.1 技能名称

技能名称采用：

- `commit-helper`

这个名称足够短，触发词自然，也能同时覆盖“生成提交信息”和“辅助自动提交”两个目标。

### 4.2 技能安装位置

该 skill 创建在当前仓库内：

- `.agents/skills/commit-helper/`

这是项目级 skill，而不是全局 skill。这样做的目的是让该仓库内的提交流程和规则与项目本身一起维护。

## 5. 触发条件

当用户表达以下意图时，应触发该 skill：

- “帮我提交”
- “生成提交信息并提交”
- “分析当前改动并提交”
- “给我一个合适的 commit message”
- “按 Conventional Commits 帮我提交”

技能应理解这些表达的共同目标是：分析当前改动、生成符合规范的中文提交信息，并在安全前提下帮助完成提交。

## 6. 默认行为

### 6.1 分析范围

默认分析整个工作区，而不是只分析已暂存改动。

这意味着技能默认面向以下改动来源：

- staged changes
- unstaged changes
- untracked files

### 6.2 自动提交流程

如果技能判断当前改动属于单一主题，则默认执行：

1. 生成中文 Conventional Commits 提交信息
2. 执行 `git add -A`
3. 执行 `git commit -m "<提交信息>"`

### 6.3 中止条件

如果技能判断改动主题混杂，则必须中止自动提交，不得擅自：

- 自动暂存部分文件
- 自动拆分多个 commit
- 自动生成一个“勉强概括全部改动”的提交

中止后应改为输出拆分建议。

## 7. 提交信息规范

### 7.1 提交格式

提交信息必须遵循 Conventional Commits：

```text
<type>(<scope>): <中文描述>
```

其中：

- `type` 使用英文关键字
- `scope` 可选，但如果能明确识别模块，优先补上
- 冒号后使用中文描述

### 7.2 允许的常见类型

技能至少应优先在这些类型中选择：

- `feat`
- `fix`
- `refactor`
- `docs`
- `style`
- `test`
- `chore`
- `perf`
- `build`
- `ci`

### 7.3 中文描述要求

中文描述必须满足：

- 简洁
- 聚焦一个核心主题
- 描述结果而不是流水账
- 避免“同时、顺便、并且”式堆叠表述

示例：

- `feat(blog): 新增博客精选文章展示区`
- `fix(auth): 修复后台登录态校验异常`
- `docs(skill): 新增中文 Conventional Commits 提交助手技能`

## 8. 单一主题与混杂改动识别规则

### 8.1 单一主题改动

满足以下大多数特征时，可视为单一主题：

- 改动围绕同一个功能、修复或重构目标
- 文件虽然分散，但属于同一条业务链路
- 可以用一句中文准确总结
- 各文件之间具备明显主从关系

典型场景包括：

- 新增一个功能及其配套组件
- 修复一个 bug 连带修改相关逻辑、样式和测试
- 一次目标明确的重构
- 围绕同一主题的文档改动

### 8.2 混杂改动

出现以下情况时，必须判定为混杂改动：

- 同时存在两个以上互不从属的目的
- 明显并列出现不同类型的大改动
- 存在独立的“顺手改一下”类改动
- 文件分布在多个独立模块，且难以用一句话概括
- 如果提交信息需要依赖“并且 / 同时 / 顺便”才能说清楚，通常就应拆分

典型场景包括：

- 页面功能改动与 `.gitignore` 调整同时存在
- 新功能和文档改动、配置改动混在一起
- 两个互不相关模块被同时修改
- 用户主动改动与生成产物混在一起

## 9. 提交流程设计

技能的默认执行流程固定为：

1. 读取工作区状态
2. 收集改动摘要
3. 判断当前改动是单一主题还是混杂改动
4. 若为单一主题：
   - 生成提交信息
   - 执行 `git add -A`
   - 执行 `git commit -m "..."`
5. 若为混杂改动：
   - 停止自动提交
   - 输出拆分建议
   - 为每组建议改动提供候选提交信息

该 skill 不应在流程中引入交互式 git。

## 10. `git add` 策略

为了避免“看似智能、实则漏文件”的问题，第一版采用简单而稳定的策略：

- 判定为单一主题时，使用 `git add -A`
- 判定为混杂改动时，不执行自动暂存

不做以下行为：

- 不自动只 add 部分文件
- 不自动跳过某些未跟踪文件
- 不自动根据文件名做主观筛选

## 11. 技能结构

技能采用最小结构：

```text
.agents/skills/commit-helper/
├── SKILL.md
└── scripts/
    └── collect_commit_context.sh
```

### 11.1 `SKILL.md`

`SKILL.md` 负责定义：

- 触发条件
- 提交流程
- 单一主题与混杂改动判断规则
- Conventional Commits 中文生成规则
- 自动提交与停止提交的边界

### 11.2 `collect_commit_context.sh`

脚本只负责采集上下文，不负责决策与提交。

脚本负责输出：

- 当前分支名
- `git status --short`
- `git diff --stat`
- staged diff 摘要
- unstaged diff 摘要
- untracked 文件列表
- 变更文件清单

脚本明确不负责：

- 不生成 commit message
- 不判断改动主题
- 不执行 `git add`
- 不执行 `git commit`

## 12. 输出格式

### 12.1 可自动提交时

当技能判断为单一主题并成功提交后，应输出：

- 当前改动主题的一句总结
- 生成的中文 Conventional Commits 提交信息
- 实际提交结果
- 最终 commit hash

### 12.2 不可自动提交时

当技能判断为混杂改动时，应输出：

- 为什么不能自动提交
- 建议拆分成哪几组
- 每组建议的提交信息

## 13. 失败处理规则

以下情况必须停止并如实反馈：

- `git add` 失败
- `git commit` 失败
- 工作区为空，没有可提交内容
- 仓库不处于可安全提交的状态
- 工作区处于 merge 冲突等高风险状态
- 改动范围不足以推断清晰主题

技能不得在失败时伪称已完成提交。

## 14. 验证方式

创建完成后，至少应验证以下 3 类场景：

### 14.1 单一主题改动

预期行为：

- skill 判断为可自动提交
- 生成中文 Conventional Commits 提交信息
- 自动执行 `git add -A` 和 `git commit`

### 14.2 混杂改动

预期行为：

- skill 判断为混杂改动
- 不执行提交
- 输出拆分建议与候选提交信息

### 14.3 无改动

预期行为：

- skill 明确说明没有可提交内容
- 不执行任何 git 写操作

## 15. 第一版实施边界

第一版 skill 只解决以下问题：

- 分析工作区改动
- 生成中文 Conventional Commits 提交信息
- 在单一主题前提下自动提交
- 在混杂改动时停止并输出拆分建议

第一版不追求处理所有高级 git 场景。

## 16. 进入实现前锁定的约束

除非后续明确修改，否则实现必须遵守以下约束：

- skill 放在仓库内 `.agents/skills/commit-helper/`
- skill 名称为 `commit-helper`
- 默认分析整个工作区
- 默认尝试自动提交
- 自动提交前提是单一主题改动
- 混杂改动时必须停止自动提交并给出拆分建议
- 提交信息必须使用中文 Conventional Commits
- 第一版使用 `SKILL.md + 轻量脚本辅助` 结构
