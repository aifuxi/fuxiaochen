# Types: Type Inference

## Why It Matters

Drizzle generates TypeScript types directly from your schema definitions. This means your
types are always in sync with your database — no codegen step, no manual type definitions.
But you need to know the right patterns to extract and use these types effectively.

## Inferring Select and Insert Types

Every Drizzle table has built-in type helpers:

```typescript
import { pgTable, integer, text, boolean, timestamp } from 'drizzle-orm/pg-core';

const users = pgTable('users', {
  id: integer().primaryKey().generatedAlwaysAsIdentity(),
  name: text('name').notNull(),
  email: text('email').notNull(),
  isActive: boolean('is_active').notNull().default(true),
  createdAt: timestamp('created_at').notNull().defaultNow(),
});

// Method 1: $inferSelect and $inferInsert on the table object
type SelectUser = typeof users.$inferSelect;
// { id: number; name: string; email: string; isActive: boolean; createdAt: Date }

type InsertUser = typeof users.$inferInsert;
// { name: string; email: string; id?: number; isActive?: boolean; createdAt?: Date }
// Note: columns with defaults or generated values become optional in insert type

// Method 2: Utility types from drizzle-orm
import { type InferSelectModel, type InferInsertModel } from 'drizzle-orm';

type SelectUser = InferSelectModel<typeof users>;
type InsertUser = InferInsertModel<typeof users>;

// Method 3: Via the _ namespace (equivalent to $inferSelect)
type SelectUser = typeof users._.$inferSelect;
type InsertUser = typeof users._.$inferInsert;
```

All three methods produce identical types. Use whichever you find most readable.

## Select vs Insert Type Differences

The key distinction is that **select types** have all fields required (they represent a
complete row from the database), while **insert types** make fields optional when they
have defaults, auto-increment, or generated values:

```typescript
type SelectUser = {
  id: number;           // required — always present in a SELECT result
  name: string;         // required
  email: string;        // required
  isActive: boolean;    // required
  createdAt: Date;      // required
};

type InsertUser = {
  id?: number;          // optional — generated by identity
  name: string;         // required — no default
  email: string;        // required — no default
  isActive?: boolean;   // optional — has .default(true)
  createdAt?: Date;     // optional — has .defaultNow()
};
```

## Using Inferred Types in Your Application

```typescript
// Function signatures
async function createUser(data: InsertUser): Promise<SelectUser> {
  const [user] = await db.insert(users).values(data).returning();
  return user;
}

// API request validation
function validateUserInput(body: unknown): InsertUser {
  // Your validation logic here
  return body as InsertUser;
}

// Component props
interface UserCardProps {
  user: SelectUser;
}
```

## Incorrect Patterns

```typescript
// Wrong: manually defining types that mirror the schema
interface User {
  id: number;
  name: string;
  email: string;
}
// This will drift out of sync when you modify the schema

// Wrong: using SelectUser for inserts
async function createUser(data: SelectUser) {
  // SelectUser requires `id` which is auto-generated
  // Use InsertUser instead
}
```

## Partial Types with Pick/Omit

When you only need certain fields:

```typescript
type UserProfile = Pick<SelectUser, 'id' | 'name' | 'email'>;
type UserUpdate = Partial<Omit<InsertUser, 'id'>>;
```
